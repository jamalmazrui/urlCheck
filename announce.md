# urlCheck Release Notes

Project home: <https://github.com/JamalMazrui/urlCheck>

`urlCheck` is free, open-source software released under the MIT License. Anyone is welcome to download and use it. The only requirement is a modern version of Windows; nothing else needs to be installed.

## Version 1.11.0

### What's new

- **Accessibility Conformance Report (`ACR.xlsx` and `ACR.docx`).** At the end of every run, urlCheck writes a draft Accessibility Conformance Report to the parent output folder, in two complementary formats: a structured workbook with one row per WCAG 2.2 success criterion (all 86 across Levels A, AA, and AAA — the obsolete 4.1.1 Parsing is omitted), and a narrative DOCX summarizing the overall picture. Both files map axe-core results to standard VPAT 2.5 terminology (Supports, Partially Supports, Does Not Support, Not Applicable, Not Evaluated). Criterion cells link to the W3C WCAG 2.2 Quick Reference. The workbook also has per-page diagnostic sheets and a Glossary sheet with key terms and external resources.
- **Conformance cell with page navigation.** The Conformance column on Sheet 1 is multi-line. Line 1 is the verdict; subsequent lines list page sheet names where the criterion was not supported (under "Not supported:") and where it could not be evaluated (under "Not evaluated:"). Only line 1 is visible by default. The user expands the row to see the page lists, then clicks a sheet tab to dig into per-page diagnostics.
- **Remarks column with axe-context labeling.** The user-editable Remarks column (formerly "Notes" — renamed to align with VPAT 2.5 "Remarks and Explanations" convention) auto-populates its first line with `Axe: fail N, pass N, incomplete N, inapplicable N` so the user has axe-context counts at a glance, while the column itself supports the user's own ACR-context remarks below.
- **Calc precedence and rollup logic.** Per-page Calc applies in this order: any-fail-with-any-pass → "partial"; any-fail-with-no-pass → "fail"; any-incomplete with no fail or pass → "manual"; any-pass with no fail or incomplete → "pass"; only-inapplicable → "na"; no axe coverage → "unknown". Across pages, the rollup uses worst-result-wins: any page failing a criterion plus any page passing it produces "Partially Supports". Counts are by node instance (one DOM element flagged), not distinct rule, summed across all included pages.
- **ACR-conformance mapping per ITI guidance.** Calc values map to VPAT 2.5 ACR terms: pass and na both become "Supports" (the criterion is satisfied vacuously when no relevant content exists, per WCAG 2.0 Understanding Conformance and ITI VPAT guidance); partial → "Partially Supports"; fail → "Does Not Support"; manual and unknown → "Not Evaluated". Note: VPAT 2.5 reserves "Not Evaluated" for AAA criteria, but urlCheck uses it more broadly because the output is a draft ACR — the user is expected to perform manual checks (listed in the Manual column) and refine the Result column before publishing.
- **Source-of-truth flag.** With `--force`, the ACR workbook is built strictly from the URLs scanned in the current session. Without `--force`, the workbook is built from every subfolder under the parent output folder that contains a `results.json` file — your accumulated scan history. The user curates the report by deleting subfolders to exclude their pages.
- **User-edit preservation.** The user-editable Result and Remarks columns on the rollup sheet survive regeneration: edits typed in one run are read back and re-applied (matched by criterion id) when the workbook is regenerated on subsequent runs.
- **Manual-test instructions for all 86 criteria.** Each WCAG 2.2 success criterion has a 4–5-step manual-test guide bundled into urlCheck, written in second-person imperative voice. Tests for Level A and AA criteria are based on Jamal Mazrui's published `WCAG_Manual_Tests.csv`; tests for Level AAA criteria were authored fresh for this release using W3C's Understanding WCAG 2.2 documents and the author's *WCAG in a Book* as the canonical content sources.
- **Spreadsheet formatting harmonized.** Both `report.xlsx` and the new `ACR.xlsx` use a consistent style: data-driven column widths capped at 50 characters with `wrap_text` enabled on overflow columns; vertical-align top; bold first row with white-on-blue fill; frozen header row; and `Title<NN>`-style named cells on each data region's column-header so screen readers announce both the column header and the row label during navigation. The Glossary sheet's multi-section layout names each section's "Term" header cell separately.
- **Log file appends by default.** The `urlCheck.log` is now opened in append mode by default, so accumulated diagnostic history across runs is preserved. Combine with `--force` (or check **Force replacements**) to replace the prior log instead. Sessions are visually separated by a blank line in append mode. The log file is written into the same folder where ACR.xlsx is written, so the user can find both side by side.
- **Logger consolidation and pre-open buffering.** Diagnostic messages emitted before the output folder is finalized are now captured in an in-memory buffer and flushed to the log file as soon as `logger.open()` is called with the resolved path. There is now a single `logger.open()` call site late in `main()`, so the log file always lands in the same folder as `ACR.xlsx` regardless of whether the output folder came from the command line, a saved configuration, or the GUI dialog. This fixes a regression where `urlCheck.log` did not appear when GUI mode was used in combination with `-l` or with a saved logging preference.
- **Accessibility failure rate.** Each per-page report (`report.htm`, `report.xlsx`) and the narrative `ACR.docx` now include a single-number summary of how well the page is doing on automated accessibility checks: `100000 × impact-weighted instance count / page bytes`, where impact weights are minor 1, moderate 2, serious 3, critical 4. The constant `100000` is tuned so the result reads naturally as a percent: typical pages land below 100%, with most well under that, while a heavily problematic page can exceed 100%. The aggregate rate for a page set is computed as sum-of-numerators over sum-of-denominators (size-weighted). Lower is better. The metric is intended to be tracked over time as the page owner remediates issues.
- **ACR.docx metadata header.** The narrative DOCX now displays the run start time, the testing engine version (axe-core), the number of pages analyzed, and the aggregate accessibility failure rate, in addition to the existing program-version line.
- **Per-violation element screenshots.** For every rule violation found by axe, urlCheck attempts to capture an element-level screenshot using axe's CSS selector. Successful captures are saved to a `violations/` subfolder as `image-001.png`, `image-002.png`, etc. The Results sheet of `report.xlsx` has a new **Image** column whose cells are clickable hyperlinks to the relative paths; clicking a cell opens the PNG in the OS default image viewer. The hyperlinks are relative, so the page subfolder can be moved, zipped, or shared freely. Where axe's selector cannot be resolved (shadow DOM, hidden elements, multiple matches, etc.), urlCheck silently skips that node — the corresponding row's Image cell is empty, and the user can still find the element using the CSS selector in the Path column.

## Version 1.10.0

### What's new

- **Edge browser handling.** urlCheck drives the system-installed Microsoft Edge through Playwright's `channel="msedge"` mechanism. The build script no longer runs `playwright install msedge`, which was actively harmful: on a Windows machine where Edge is already installed (the universal case), Playwright errors with "msedge is already installed on the system" and would fail the build. Edge ships in-box with Windows 10 and 11.
- **Friendly Edge error.** If the rare situation occurs where Playwright cannot find Edge (Edge removed by the user, unusual configuration), urlCheck now surfaces a friendly message pointing at https://www.microsoft.com/edge instead of a Python traceback.
- **64-bit build guard.** `buildUrlCheck.cmd` verifies the selected Python interpreter is 64-bit before invoking PyInstaller. A 32-bit Python is rejected with a clear error so the produced `urlCheck.exe` is consistently 64-bit.
- **Skip-or-overwrite output behavior, consistent with extCheck and 2htm.** When a per-page output folder with the same sanitized page title already exists, urlCheck now skips that url by default rather than auto-suffixing a new folder. With `--force` (or the Force replacements checkbox in the dialog), the existing folder is emptied and a fresh scan is written into it. The skip decision is made right after the page title is read, before the expensive accessibility scan, so re-running urlCheck on a long url list is cheap when most pages have already been scanned. Note: this is a behavior change from prior versions, which used auto-suffixed folder names like `My Page-001`.
- **Force replacements checkbox in the GUI dialog.** Matches the corresponding checkbox in extCheck and 2htm.
- **Picker initial folder.** The Browse source and Choose output buttons now open at the folder derived from the text-field value when that value points to an existing path (whether the user just typed it or it was loaded from a saved configuration), and at the user's Documents folder otherwise. The folder picker honors the initial path via a BFFCALLBACK that posts BFFM_SETSELECTION on BFFM_INITIALIZED — the previous version was unable to honor an initial path with the classic SHBrowseForFolder style.
- **Friendlier source-field parsing.** When you supply a single URL list file whose path contains spaces, you no longer need to put quotes around it. urlCheck tests the entire trimmed source field as a path first; only when it is not an existing file does it fall back to space-tokenization (treating the tokens as urls or domains). Quotes are only required when supplying multiple specs and at least one contains a space.
- **Source field accessibility.** The Source urls and Output folder text boxes now have explicit `AccessibleName` properties so JAWS and NVDA reliably announce each field by its label when focus moves to it, regardless of label-textbox visual layout.
- **Cleaner result messages.** The result dialog in GUI mode and the matching console output in CLI mode now show only what the user needs: per-url status and the structured summary. Diagnostic writes no longer carry an `[ERROR]` prefix; the level data is recorded in the log file when `-l` is given. The no-source-given path simply prints `No urls to scan.` rather than a program-name-and-usage description.
- **Output-folder create prompt.** If you press OK with an output folder that does not yet exist, urlCheck asks "Create [path]?" with default Yes. Choosing No keeps the dialog open with focus on the output field.
- **Source label and lowercase "url".** The dialog field label is now **Source urls** (was Source files), and "url" is rendered lowercase throughout the help text, error messages, and documentation. The exception is camelCase identifiers like `urlCheck` and `getNormalizedUrl` where the casing is structural.
- **Invisible mode label.** The Invisible checkbox in the dialog is labeled "Invisible mode" (mnemonic on the I).
- **Authenticate credentials.** New `-a` / `--authenticate` parameter (and matching **&Authenticate credentials** dialog checkbox). When active, urlCheck pauses after each newly-encountered **registrable domain** loads -- the first time it sees a given site in this run -- and prompts the user to authenticate (sign in, accept cookies, dismiss popups, complete 2FA) and press Enter (CLI) or click OK (GUI) to resume. Subdomains collapse to the registrable domain (so `m.facebook.com`, `www.facebook.com`, and `accounts.facebook.com` all share one prompt under `facebook.com`); a small set of public 2-level suffixes is recognized so `bbc.co.uk` is not collapsed to `co.uk`. `file://` and `about:` urls never prompt. The CLI prompt is `"Authenticate credentials, if needed, to show the page. To resume automation, press Enter:"`; the GUI uses a top-most owned MessageBox so the dialog appears on top of the Edge browser window. After confirmation, urlCheck waits for the page to settle (load event + networkidle + 4-second extra delay specific to `-a` runs) before running the accessibility scan, since post-auth SPA UIs (Facebook feed, WhatsApp main panel, Slack workspace) render their main content asynchronously after networkidle fires; urlCheck also swaps to the most-recently-active page in the browser context so OAuth/SSO popup and redirect flows are handled correctly. By default, urlCheck launches Edge with a fresh **temporary** profile (your real Edge profile is not exposed to the scanned site) and disconnects Playwright's automation channel from Edge during the user-interaction pause -- via Playwright's documented `chromium.connect_over_cdp` API, attaching to msedge.exe launched as a subprocess with `--remote-debugging-port=0 --user-data-dir=<temp>`. The disconnect/reconnect pattern improves the chance of success against sites that detect active automation (WhatsApp Web is a notable example). To use your real Edge profile instead, see the new **Main profile** option below. If both `--authenticate` and `--invisible` are set (CLI) or both checkboxes are checked (GUI), urlCheck overrides `--invisible` at run time and launches with a visible browser (an auth prompt needs a visible window); the override is logged. Suppresses Playwright's `--enable-automation` switch (so the 'controlled by automated test software' infobar does not appear), enables Chromium's sandbox (so the 'unsupported flag: --no-sandbox' warning does not appear), and overrides `navigator.webdriver` to undefined via a context init script. The setting is persisted by `Use configuration` and shown in the log header.
- **Main profile.** New `-m` / `--main-profile` parameter (and matching **&Main profile** dialog checkbox). When set, urlCheck launches Edge with your real (default) Edge user profile at `%LOCALAPPDATA%\Microsoft\Edge\User Data` so saved logins, cookies, and session state are available to the scanned page -- if you are already signed into a site in your normal Edge, you remain signed in inside urlCheck's Edge session. Without `-m`, urlCheck uses a fresh ephemeral temporary profile so the scan is anonymous and your real profile is not exposed to the scanned site. Independent of `-a`: `-m` alone uses the real profile with no auth prompts (useful for scanning sites you are already signed into, fully unattended); `-m -a` uses the real profile WITH auth prompts (useful when partial sign-in needs interactive completion); `-a` alone uses a temporary profile WITH auth prompts and the disconnect pattern (the strongest anti-detection mode). Whenever `-m` is set, urlCheck checks for a running `msedge.exe` process and refuses to proceed: the CLI exits with a friendly message asking the user to close all Edge windows; the GUI shows a dialog explaining why it cannot proceed and returns focus to where it was so the user can close Edge and submit the dialog again without restarting urlCheck. Edge cannot share a profile across two processes. With `-m -a`, the disconnect/reconnect pattern is NOT used (Chrome 136+ refuses `--remote-debugging-port` against the default user-data-dir for security); the auth prompt fires while Playwright remains attached. The setting is persisted by `Use configuration` and shown in the log header.
- **Structured results summary.** The per-url run is now summarized in three sections — `Scanned N url(s):`, `Failed to scan N url(s):`, and `Skipped N url(s). Check "Force replacements" to overwrite.` — each shown only when its count is non-zero, with singular "url" / plural "urls" inflection. Failed entries include a short reason after the url when one is available (`https://example.com: Timeout`); the full traceback goes to the log when `-l` is on.
- **CLI vs GUI output styles.** In CLI mode the URLs print inline as the loop runs — natural progress feedback for the console user. In GUI mode the loop is silent on stdout (urlCheck has no per-page status form), and the structured summary is the final dialog. The structured summary is printed in both modes, but the per-url lists are suppressed in CLI mode (they would just repeat what already scrolled by).
- **No implicit error files.** Earlier versions wrote an `error.txt` and a fallback `untitled-page` folder when a scan crashed unexpectedly. urlCheck now creates files only when the user explicitly asks for them. Errors are surfaced to the console (always) and to `urlCheck.log` when `-l` is on. If you run without `-l` and want full tracebacks for any errors, the run summary reminds you to add `-l` next time.
- **Log header.** When `-l` (Log session) is enabled, the log file now begins with a clean header before the timestamped processing notifications: program name and version, a friendly run timestamp (`Run on May 1, 2026 at 2:30 PM`), and a `Parameters:` block listing each setting with both explicit and defaulted values resolved (Source urls, Output folder, Force replacements, Invisible mode, View output, Use configuration, Log session, GUI mode, Working folder, Command line). The header is followed by the normal timestamped log entries.
- **Camel Type coding standard.** All variable names follow the project's Camel Type style. The `o` prefix is reserved for COM objects only; non-COM objects use the lowercase class name as their prefix. Examples: `oPath` → `path`, `oResponse` → `response`, `oDlg` → `dialog`, `oError` → `ex`. urlCheck does not use COM, so it has no remaining `o`-prefixed variables.
- **Cross-program naming.** Identifier names for shared concepts now match across the three companion tools (urlCheck, extCheck, 2htm). The program-name and version constants are `sProgramName` and `sProgramVersion`; the config and log filename constants are `sConfigDirName`, `sConfigFileName`, `sLogFileName`; the source and output-folder variables are `sSource` and `sOutputDir`; the GUI layout constants are all `iLayout*` (left, right, top, gap, rowGap, labelWidth, buttonWidth, buttonHeight, textHeight, formWidth). The `logger` class now has the same surface in all three programs: `open`, `close`, `info`, `warn`, `error`, `debug`.

### Technical notes

- The icon is embedded in `urlCheck.exe` at build time via PyInstaller's `--icon` flag (multi-resolution: 16, 24, 32, 48, 64, 128, 256). Shortcuts inherit the icon.
- Two ctypes Structure class definitions (`ProcessEntry32` and `BrowseInfoW`) were renamed from their old `o`-prefixed forms to follow Python's PascalCase class-naming convention.
- The build script (`buildUrlCheck.cmd`) writes `urlCheck.exe` to the current working folder rather than a `dist\` subfolder, and it no longer copies `urlCheck.cmd` or `urlCheck.ico` aside (those files already live in the working folder next to `urlCheck.iss`). PyInstaller's intermediate work files are stashed in a `build\` subfolder out of the way; the auto-generated `urlCheck.spec` is removed after a successful build. The script uses no forward `call :label`; it ships with CRLF line endings as defense-in-depth.

### Installer (`urlCheck_setup.exe`)

- 64-bit only.
- Prompts for the installation folder on every run (default: `C:\Program Files\urlCheck`). The folder page is now explicitly enabled (`DisableDirPage=no` in the .iss); previously it was at the Inno Setup default of `auto`, which silently skipped the page on reinstalls of the same `AppId`. The previous folder is pre-filled, so on a reinstall the user just presses Next to keep the same path.
- Includes a brief MIT-license summary on the welcome page.
- Installs only HTML versions of the documentation (`ReadMe.htm`, `Announce.htm`, `License.htm`); the Markdown counterparts and source/build/installer scripts live in the GitHub repository.
- The "Launch urlCheck now" checkbox on the final page reminds the user that the desktop hotkey is Alt+Ctrl+U.
