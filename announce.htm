urlCheck Announce

urlCheck Release Notes

Version 1.10.0

What’s new

  • Edge browser handling. urlCheck drives the system-installed Microsoft Edge through Playwright’s channel="msedge" mechanism. The build script no longer runs playwright install msedge, which was actively harmful: on a Windows machine where Edge is already installed (the universal case), Playwright errors with “msedge is already installed on the system” and would fail the build. Edge ships in-box with Windows 10 and 11.

  • Friendly Edge error. If the rare situation occurs where Playwright cannot find Edge (Edge removed by the user, unusual configuration), urlCheck now surfaces a friendly message pointing at https://www.microsoft.com/edge instead of a Python traceback.

  • 64-bit build guard. buildUrlCheck.cmd verifies the selected Python interpreter is 64-bit before invoking PyInstaller. A 32-bit Python is rejected with a clear error so the produced urlCheck.exe is consistently 64-bit.

  • Skip-or-overwrite output behavior, consistent with extCheck and 2htm. When a per-page output folder with the same sanitized page title already exists, urlCheck now skips that url by default rather than auto-suffixing a new folder. With --force (or the Force replacements checkbox in the dialog), the existing folder is emptied and a fresh scan is written into it. The skip decision is made right after the page title is read, before the expensive accessibility scan, so re-running urlCheck on a long url list is cheap when most pages have already been scanned. Note: this is a behavior change from prior versions, which used auto-suffixed folder names like My Page-001.

  • Force replacements checkbox in the GUI dialog. Matches the corresponding checkbox in extCheck and 2htm.

  • Picker initial directory. The Browse source and Choose output buttons now open at the directory derived from the text-field value when that value points to an existing path (whether the user just typed it or it was loaded from a saved configuration), and at the user’s Documents folder otherwise. The folder picker honors the initial path via a BFFCALLBACK that posts BFFM_SETSELECTION on BFFM_INITIALIZED — the previous version was unable to honor an initial path with the classic SHBrowseForFolder style.

  • Friendlier source-field parsing. When you supply a single URL list file whose path contains spaces, you no longer need to put quotes around it. urlCheck tests the entire trimmed source field as a path first; only when it is not an existing file does it fall back to space-tokenization (treating the tokens as urls or domains). Quotes are only required when supplying multiple specs and at least one contains a space.

  • Source field accessibility. The Source urls and Output directory text boxes now have explicit AccessibleName properties so JAWS and NVDA reliably announce each field by its label when focus moves to it, regardless of label-textbox visual layout.

  • Cleaner result messages. The result dialog in GUI mode and the matching console output in CLI mode now show only what the user needs: per-url status and the structured summary. Diagnostic writes no longer carry an [ERROR] prefix; the level data is recorded in the log file when -l is given. The no-source-given path simply prints No urls to scan. rather than a program-name-and-usage description.

  • Output-directory create prompt. If you press OK with an output directory that does not yet exist, urlCheck asks “Create [path]?” with default Yes. Choosing No keeps the dialog open with focus on the output field.

  • Source label and lowercase “url”. The dialog field label is now Source urls (was Source files), and “url” is rendered lowercase throughout the help text, error messages, and documentation. The exception is camelCase identifiers like urlCheck and getNormalizedUrl where the casing is structural.

  • Invisible mode label. The Invisible checkbox in the dialog is labeled “Invisible mode” (mnemonic on the I).

  • Authenticate credentials. New -a / --authenticate parameter (and matching &Authenticate credentials dialog checkbox). When active, urlCheck launches Edge against your real Edge user profile (%LOCALAPPDATA%\Microsoft\Edge\User Data) rather than a fresh ephemeral profile, so cookies, saved logins, and session state are available to the scanned page. If you are already signed into a site in your normal Edge browser, you are still signed in inside urlCheck’s Edge session. The browser must not be running when urlCheck starts: Edge cannot share a profile across two processes, so urlCheck checks for a running msedge.exe first and asks you to close all Edge windows before retrying. To minimize problems with sites that detect browser automation (Facebook, WhatsApp Web, etc.), the -a launch suppresses Playwright’s --enable-automation switch (so the “Microsoft Edge is being controlled by automated test software” infobar does not appear), enables Chromium’s sandbox (so Edge does not show the “unsupported command-line flag: –no-sandbox” warning bar that Playwright’s default disabled-sandbox configuration produces), and overrides navigator.webdriver to undefined via a context init script that runs on every new document before any site JavaScript reads it. After Edge launches, urlCheck pauses after each page first loads — the first time it sees a given registrable domain in the run — to give you a chance to interact with the browser if needed (sign in, dismiss cookie banners, accept popups, complete 2FA) and confirm when ready to proceed. The pause uses the console in CLI mode (the prompt is preceded by a newline so it doesn’t run into the inline-progress url; “Authenticate credentials, if needed, to show the page. To resume automation, press Enter:” then waits for Enter) and a top-most owned MessageBox in GUI mode (the dialog appears on top of the Edge browser window rather than behind it, so screen readers can find and announce it). Subsequent urls on the same registrable domain reuse the established session without prompting. After confirmation, urlCheck waits for the page to settle (load event, then networkidle, then a 4-second extra delay specific to -a runs) before running the accessibility scan, since post-auth single-page-app UIs (Facebook feed, WhatsApp main panel, Slack workspace) render their main content asynchronously after networkidle fires. urlCheck swaps to the most-recently-active page in the browser context (so OAuth/SSO popups, post-login redirects, and “logged in to a different page” flows are handled correctly). Subdomains collapse to the registrable domain by default (so m.facebook.com, www.facebook.com, and accounts.facebook.com are all treated as facebook.com); a small set of public 2-level suffixes is recognized so bbc.co.uk is not collapsed to co.uk. file:// and about: urls never prompt. Forces visible browser mode for the run when invisible mode is also set (logged as an override). The setting is persisted by Use configuration and shown in the log header.

  • Temporary profile with disconnect/reconnect. New -t / --temp-profile parameter, used together with -a. When set, urlCheck launches msedge.exe directly as a subprocess against a fresh temporary profile (a randomly-named directory under %TEMP%) rather than using your real Edge user profile, and uses Playwright’s chromium.connect_over_cdp() API to attach and detach during each url’s authentication pause. Specifically: at run start, urlCheck launches msedge.exe --remote-debugging-port=0 --user-data-dir=<temp>, waits for Edge to write its chosen port to the DevToolsActivePort file in the user-data-dir, and connects via chromium.connect_over_cdp("http://localhost:<port>"). When the authentication prompt fires, urlCheck calls browser.close() on the connected client — this severs Playwright’s CDP connection without killing the Edge subprocess (urlCheck owns the Edge process via subprocess.Popen, not Playwright), so the user can interact with the visible browser window while no automation channel is attached. After the user confirms (Enter in CLI mode, OK in GUI mode), urlCheck reconnects via connect_over_cdp to the same endpoint, finds the page the user was last on (which may have navigated through OAuth/SSO/etc.), reapplies the navigator.webdriver init script to the newly-bound context, and proceeds with the scan. Cookies and session state set during authentication persist within the same Edge process via its in-memory cookie jar and on disk in the temp user-data-dir, so subsequent urls on the same domain reuse the established session without re-prompting — exactly like normal browsing. This trades saved logins (the profile is fresh — every site that requires authentication needs fresh credentials) for stronger anti-detection: during user interaction, the browser truly appears uncontrolled to the page, which can let highly-fingerprinted sites complete navigations they refuse with the default-profile path (WhatsApp Web is a notable example, though success is not guaranteed even with this flag). Has no effect without -a. The Edge subprocess is killed and the temp directory is removed at end of run. Note: Chrome 136+ refuses --remote-debugging-port against the default Chrome user-data directory for security reasons, which is why this flag exists separately from -a rather than being available against your real Edge profile. Note also that playwright-python (unlike playwright-node) does not expose BrowserType.launch_server; connect_over_cdp is the documented Playwright Python API for the launch-it-ourselves-then-attach pattern.

  • Structured results summary. The per-url run is now summarized in three sections — Scanned N url(s):, Failed to scan N url(s):, and Skipped N url(s). Check "Force replacements" to overwrite. — each shown only when its count is non-zero, with singular “url” / plural “urls” inflection. Failed entries include a short reason after the url when one is available (https://example.com: Timeout); the full traceback goes to the log when -l is on.

  • CLI vs GUI output styles. In CLI mode the URLs print inline as the loop runs — natural progress feedback for the console user. In GUI mode the loop is silent on stdout (urlCheck has no per-page status form), and the structured summary is the final dialog. The structured summary is printed in both modes, but the per-url lists are suppressed in CLI mode (they would just repeat what already scrolled by).

  • No implicit error files. Earlier versions wrote an error.txt and a fallback untitled-page folder when a scan crashed unexpectedly. urlCheck now creates files only when the user explicitly asks for them. Errors are surfaced to the console (always) and to urlCheck.log when -l is on. If you run without -l and want full tracebacks for any errors, the run summary reminds you to add -l next time.

  • Log header. When -l (Log session) is enabled, the log file now begins with a clean header before the timestamped processing notifications: program name and version, a friendly run timestamp (Run on May 1, 2026 at 2:30 PM), and a Parameters: block listing each setting with both explicit and defaulted values resolved (Source urls, Output directory, Force replacements, Invisible mode, View output, Use configuration, Log session, GUI mode, Working directory, Command line). The header is followed by the normal timestamped log entries.

  • Camel Type coding standard. All variable names follow the project’s Camel Type style. The o prefix is reserved for COM objects only; non-COM objects use the lowercase class name as their prefix. Examples: oPath → path, oResponse → response, oDlg → dialog, oError → ex. urlCheck does not use COM, so it has no remaining o-prefixed variables.

  • Cross-program naming. Identifier names for shared concepts now match across the three companion tools (urlCheck, extCheck, 2htm). The program-name and version constants are sProgramName and sProgramVersion; the config and log filename constants are sConfigDirName, sConfigFileName, sLogFileName; the source and output-directory variables are sSource and sOutputDir; the GUI layout constants are all iLayout* (left, right, top, gap, rowGap, labelWidth, buttonWidth, buttonHeight, textHeight, formWidth). The logger class now has the same surface in all three programs: open, close, info, warn, error, debug.

Technical notes

  • The icon is embedded in urlCheck.exe at build time via PyInstaller’s --icon flag (multi-resolution: 16, 24, 32, 48, 64, 128, 256). Shortcuts inherit the icon.
  • Two ctypes Structure class definitions (ProcessEntry32 and BrowseInfoW) were renamed from their old o-prefixed forms to follow Python’s PascalCase class-naming convention.
  • The build script (buildUrlCheck.cmd) writes urlCheck.exe to the current working directory rather than a dist\ subfolder, and it no longer copies urlCheck.cmd or urlCheck.ico aside (those files already live in the working directory next to urlCheck.iss). PyInstaller’s intermediate work files are stashed in a build\ subfolder out of the way; the auto-generated urlCheck.spec is removed after a successful build. The script uses no forward call :label; it ships with CRLF line endings as defense-in-depth.

Installer (urlCheck_setup.exe)

  • 64-bit only.
  • Prompts for the installation directory on every run (default: C:\Program Files\urlCheck). The directory page is now explicitly enabled (DisableDirPage=no in the .iss); previously it was at the Inno Setup default of auto, which silently skipped the page on reinstalls of the same AppId. The previous directory is pre-filled, so on a reinstall the user just presses Next to keep the same path.
  • Includes a brief MIT-license summary on the welcome page.
  • Installs only HTML versions of the documentation (ReadMe.htm, Announce.htm, License.htm); the Markdown counterparts and source/build/installer scripts live in the GitHub repository.
  • The “Launch urlCheck now” checkbox on the final page reminds the user that the desktop hotkey is Alt+Ctrl+U.
